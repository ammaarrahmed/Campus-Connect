# Good First Issue Strategy

Beginner-friendly issues help new contributors onboard quickly and build confidence.

## What Qualifies

A `good first issue` should:

- Have a clear problem statement
- Be scoped to 1-3 files when possible
- Avoid deep architecture changes
- Include acceptance criteria and test expectations
- Have at least one maintainer available for guidance

## Labeling Model

Use these labels together:

- `good first issue`
- `help wanted`
- area label (example: `documentation`, `bug`, `enhancement`)

## Authoring Template for Beginner Issues

Use this structure when opening beginner tasks:

1. Context: why this matters
2. Current behavior
3. Target behavior
4. Acceptance criteria
5. Suggested files to inspect
6. Testing instructions

## Example Beginner Tasks

- Improve empty-state copy in a single screen
- Add unit tests for pure model/provider logic
- Fix typo or broken doc links
- Add missing loading/error state in one widget

## Maintainer Best Practices

- Respond quickly to claim comments
- Offer implementation hints without over-prescribing
- Prefer review coaching over force-pushing fixes
- Celebrate first merged contributions in release notes
